# IPNL Paper Pilot v0.0.2 P0 Architecture: Mentor Brain + Polish/Check Engine

Status: P0 refocused architecture / recommended baseline for the next implementation round
Audience: project owner, Codex operators, future implementers
Scope: only the first two modules that matter most right now — `mentor-brain` and the `polish/check engine`
Position: this version is optimized for the real pain point — “the research is done, but the mature paper still does not meet the supervisor’s standard”

---

## 0. Why this document is being rewritten

The v0.0.1 direction was broadly correct, but it still had three structural issues:

1. **The first version was still too wide.**
   The full lane architecture was coherent, but the shortest value path is not end-to-end autonomous paper writing. It is rescuing near-finished papers first.

2. **The mapping from Codex capability to product architecture was still too coarse.**
   The earlier document separated skills, agents, and scripts well, but it did not make hooks, runtime state, artifact persistence, and native Codex/OMX capabilities explicit enough.

3. **P0 was not derived aggressively enough from the user’s real bottleneck.**
   The current bottleneck is not “I cannot generate a paper.” The real bottleneck is:
   - the idea exists,
   - the experiments are done quickly,
   - the research content is already there,
   - but the paper still feels too AI-like, too machine-like, too soft, too imprecise, and too unlike the supervisor.

Therefore, **v0.0.2 redefines the first product as a supervisor-brain-driven revision system for mature drafts**.

In one sentence:

> Do not overbuild full autonomous paper writing first. First build a system that distills the supervisor’s brain and uses it to diagnose, rank, and rewrite mature drafts.

## 1. Real user scenario and first success definition

### 1.1 Real user scenario

The user is not a generic graduate student. The actual profile is more specific:

- works in GNSS / navigation / deep learning / sensor fusion style research;
- can do the research and run the experiments;
- already has a relatively mature draft;
- the main bottleneck is not missing content, but writing that repeatedly fails the supervisor’s standard;
- supervisor feedback concentrates on:
  - AI flavor,
  - imprecise terminology,
  - translated or machine-like sentences,
  - weak paragraph rhythm,
  - hard transitions,
  - weak or unstable claims,
  - wrong emphasis order,
  - contributions and conclusions that do not sound like the supervisor.

### 1.2 What P0 must solve

P0 is not meant to solve every problem in writing a full paper from scratch.

It is meant to solve this:

> Given a mature draft plus supervisor materials plus chats/comments/revision traces, can the system act like a supervisor-side guide that helps Codex find the real problems, prioritize them, reduce AI flavor, and revise the paper toward supervisor-level quality?

### 1.3 First success criteria

P0 is worth continuing if these four things become true:

1. the system can distill a reusable **supervisor writing protocol** from PDF/TXT/chat/transcript materials;
2. the system can produce a useful **diagnostic report** for a mature paper rather than vague advice;
3. the system can generate a **supervisor-style revision agenda** and a revised version that is noticeably closer to the supervisor’s expectations;
4. the user experience improves materially: revision becomes directed and concentrated instead of painful and open-ended for months.

## 2. P0 scope freeze: dual core only, not full stack

### 2.1 P0 only goes deep on two modules

P0 should only fully build these two modules:

1. **`mentor-brain`**
   - learns how the supervisor judges, writes, rewrites, and tightens;
   - stores terminology preferences, disliked phrasing, paragraph movement, and revision patterns;
   - does not write the current paper directly; it supplies supervisor intelligence.

2. **`polish/check engine`**
   - ingests a mature draft;
   - checks AI flavor, wording, rhythm, structure, transitions, terminology, and claim/evidence expression risks;
   - produces a revision agenda;
   - drives Codex to revise the draft section by section and sentence by sentence.

### 2.2 P0 keeps only a lightweight support brain

P0 does **not** need the full `lab-dive` pipeline yet, but it still needs a **`project-context-lite`** layer:

- extracts minimal project facts from the mature draft, figures, method, conclusion, and notes;
- prevents mentor-style revision from damaging scientific meaning;
- acts as a safety rail, not as a full research-understanding system.

### 2.3 Explicit non-goals

P0 explicitly does not attempt:

- full paper generation from scratch;
- full reviewer simulation;
- multi-mentor distillation;
- background continuous learning;
- domain-general academic writing;
- black-box one-click rewriting without artifacts.

## 3. What OpenAI Codex / OMX can actually do for this project

This section matters because the architecture must be grounded in what Codex is genuinely good at.

### 3.1 Capability map

| Capability layer | Available ability | Correct use in this project | P0 critical? |
|---|---|---|---|
| Long-context synthesis and rewriting | read long documents, compare versions, integrate multiple sources | distill mentor brain, inspect mature drafts, produce revision agendas | Yes |
| Local file and repo inspection | inspect directories, read files, run safe shell commands | collect mentor materials, read drafts, build artifacts | Yes |
| Verifiable execution | run scripts, export reports, serialize outputs | manifests, diff reports, deterministic checks | Yes |
| Workflow orchestration | maintain process via skills / plans / state | make revision repeatable and traceable | Yes |
| Multi-agent decomposition | planner / architect / executor / critic / verifier roles | split diagnosis, rewriting, and verification when needed | Secondary |
| External retrieval | Context7 / web / official docs | fetch necessary technical or tooling references | Secondary |
| Persistent memory and state | `.omx/state`, notepad, project memory | preserve paper-specific rules and lessons | Yes |
| Hooks / notifications / HUD | omx hooks / tmux-hook / notify / hud | auto-save artifacts, checkpoint long runs, notify on completion | Yes |
| Text/code diagnostics | shell, regex, diff, structured checks | terminology consistency, pattern scans, before/after comparisons | Yes |

### 3.2 The most valuable Codex capabilities in P0

The highest-value capabilities are not “automatic writing.” They are:

1. reading many heterogeneous supervisor materials and compressing them into an executable writing protocol;
2. producing fine-grained diagnostics on a mature draft rather than generic editing advice;
3. rewriting concrete sentences and paragraphs while preserving global style consistency;
4. preserving the revision process as inspectable artifacts.

### 3.3 What Codex should not be used for

Codex should not be misused to:

- guess supervisor style without evidence;
- polish language while ignoring claim integrity;
- rewrite invisibly without artifacts;
- replace thinking with oversized scripts.

### 3.4 Runtime layering rule

1. **Skills manage workflow, not judgment.**
2. **Agents perform diagnosis, ranking, and rewriting.**
3. **Scripts provide deterministic scaffolding.**
4. **Hooks only trigger or persist — they do not reason.**

## 4. P0 core system: dual core plus two guardrails

### 4.1 System overview

P0 should expose only four explicit layers:

1. **Mentor Brain Core**
2. **Draft Xray & Revision Engine**
3. **Project Context Lite**
4. **Verification & Artifact Guardrails**

- `Mentor Brain Core` answers how the supervisor thinks and writes.
- `Draft Xray & Revision Engine` answers what is wrong with the mature draft and how to fix it.
- `Project Context Lite` protects scientific meaning.
- `Verification & Artifact Guardrails` keep the workflow inspectable and debuggable.

### 4.2 Mentor Brain Core

**Mission**: distill a supervisor writing judgment system from trusted supervisor materials.

**Inputs**:
- supervisor papers (PDF/TXT)
- supervisor-edited student drafts when available
- transcript/chat/commentary materials
- revision traces
- a small number of aligned high-quality reference papers

**Outputs**:
- `mentor_material_manifest`
- `mentor_corpus_index`
- `mentor_writing_protocol`
- `mentor_rewrite_patterns`
- `mentor_rejection_lexicon`
- `mentor_terminology_preferences`
- `compiled_mentor_profile`

**Must answer**:
- How does the supervisor structure a paragraph?
- Which expressions does the supervisor reject?
- How does the supervisor turn weak sentences into strong ones?
- How does the supervisor frame abstracts, contributions, and conclusions?
- Which terminology, tone, force, and rhythm does the supervisor prefer?

**Must not do**:
- validate current project truth;
- directly rewrite the current target paper;
- learn from the target draft itself.

### 4.3 Draft Xray & Revision Engine

**Mission**: perform a supervisor-like diagnostic plus a supervisor-like revision pass on a mature draft.

**Internal submodules**:
1. `draft-intake`
2. `paper-xray`
3. `issue-ranking`
4. `truth-check-lite`
5. `mentor-polish`
6. `revision-verify`

**Core questions**:
- Where does the paper feel most AI-like?
- Which problems are actually framing problems rather than language problems?
- Which sentences are weak, vague, or underpowered?
- Which paragraphs dump information rather than tell a controlled story?
- Which terminology and transitions diverge from supervisor habits?
- Which revisions improved the draft, and which merely changed wording?

### 4.4 Project Context Lite

**Mission**: provide a lightweight project brain that protects scientific meaning without requiring the full `lab-dive` system.

**Minimal outputs**:
- `project_fact_sheet`
- `claim_evidence_minimap`
- `terminology_ground_truth`
- `do_not_break_constraints`

### 4.5 Verification & Artifact Guardrails

Every meaningful run must emit:
- input manifest,
- diagnostic report,
- revision agenda,
- before/after diff,
- unresolved risks,
- suggested next pass.

## 5. Two-brain boundary rules

### 5.1 What Mentor Brain may learn from

Allowed sources:
- supervisor-authored papers,
- supervisor speech/comments/chats,
- supervisor revision traces,
- supervisor-approved expression patterns,
- a small number of aligned reference papers.

### 5.2 What Mentor Brain must never learn from

Forbidden sources:
- the current target paper,
- poor student drafts,
- unknown AI-generated writing,
- generic “fancy” templates that are not aligned with the supervisor.

### 5.3 What Project Context Lite may contain

- core project facts,
- figures, experiments, captions, and setup,
- user-provided explanations,
- claim-evidence bindings.

### 5.4 Why this boundary is non-negotiable

Because the actual need is not another general AI writer.

The need is:

> a system that can stably separate “supervisor style” from “current project truth.”

If those are mixed, the system becomes neither faithful nor useful.

## 6. P0 workflow derived from the real bottleneck

### Stage A — Mentor material ingestion and cleaning

Priority order:
1. supervisor-authored papers;
2. supervisor revision traces;
3. chats, transcripts, comments;
4. a small reference set.

Outputs:
- source inventory,
- trust levels,
- extracted text,
- normalized material packet.

### Stage B — Mentor brain distillation

Required outputs:
- terminology glossary,
- rejected expression list,
- section-wise writing preferences,
- sentence rewrite patterns,
- paragraph movement patterns,
- claim-strengthening patterns.

### Stage C — Mature draft intake

Required extraction:
- section map,
- title and abstract,
- key contribution sentences,
- core figures,
- method core,
- experiment takeaway paragraphs,
- conclusion,
- suspected AI-hotspot regions.

### Stage D — Paper Xray

The first version should always score eight dimensions:
1. `mentor-drift`
2. `ai-smell`
3. `terminology-risk`
4. `claim-precision`
5. `transition-rhythm`
6. `paragraph-architecture`
7. `evidence-language-gap`
8. `truth-risk-lite`

Outputs:
- `xray_report`
- `issue_board`
- `severity_ranking`
- `rewrite_candidates`

### Stage E — Revision agenda generation

Priority order should be:
1. problems that may damage scientific meaning,
2. problems the supervisor would reject immediately,
3. the strongest AI-flavor problems,
4. ordinary polish issues.

Outputs:
- `revision_agenda.md`
- `section_priority_map.json`
- `do_first_do_not_touch.md`

### Stage F — Mentor Polish execution

Use four layers:
1. macro pass,
2. paragraph pass,
3. sentence pass,
4. terminology pass.

### Stage G — Post-revision verification

Must answer:
- which issues were truly fixed,
- which only changed surface wording,
- where scientific meaning may have drifted,
- which decisions still require user/supervisor approval.

Outputs:
- `revision_delta_report`
- `remaining_risks`
- `mentor_alignment_scorecard`

## 7. Detailed allocation: Skill vs Agent vs Script vs Hook

## 7.1 Skills: workflow and governance

Skills define:
- when a lane is entered,
- required input shape,
- output artifact contract,
- retry logic,
- handoff logic,
- done criteria.

### Recommended P0 skills

| Skill | Role | Inputs | Outputs | P0 required? |
|---|---|---|---|---|
| `mentor-brain` | mentor material distillation entrypoint | PDF/TXT/chat/transcript | mentor profile / protocol / lexicon | Yes |
| `paper-xray` | mature draft diagnosis entrypoint | mature draft + mentor profile | xray report / issue board | Yes |
| `mentor-polish` | supervisor-style revision entrypoint | mature draft + agenda + mentor profile | revised draft / revision notes | Yes |
| `truth-check-lite` | scientific-meaning safety pass | revised draft + fact sheet | warnings / lite verification | Yes |
| `reference-mirror` | paragraph-level comparison to strong references | weak paragraph + strong refs | contrast packet | Recommended |
| `draft-compare` | before/after interpretation | before and after drafts | delta report | Recommended |

Principles:
1. skills define process, not hidden rewriting;
2. every skill requires artifacts;
3. every skill must specify stop conditions;
4. skills must serve the mature-draft rescue scenario first.

## 7.2 Agents: the actual thinkers and revisers

P0 should reuse current native Codex roles instead of inventing unnecessary pseudo-agents.

| Agent type | Responsibility in this project | P0 usage |
|---|---|---|
| `planner` | builds revision agendas and pass ordering | agenda generation |
| `architect` | protects boundaries and lane responsibilities | structure / guardrail review |
| `executor` | performs actual rewriting | main P0 revision worker |
| `critic` | finds AI flavor and supervisor-disliked weaknesses | diagnosis / challenge pass |
| `verifier` | checks whether revisions really improved the draft | post-polish validation |
| `researcher` | fetches external references when needed | occasional support |
| `writer` | summarizes rationale and artifacts cleanly | user-facing reporting |

Recommended multi-step pattern when complexity rises:
1. `critic` identifies the real problems,
2. `executor` rewrites with mentor brain guidance,
3. `verifier` checks for regressions or drift.

## 7.3 Scripts: deterministic workstations

Scripts do not think. They stabilize inputs and outputs.

| Proposed script | Purpose | Why it matters |
|---|---|---|
| `scripts/build_mentor_manifest.py` | build source inventory and trust labels | avoids messy mentor inputs |
| `scripts/extract_pdf_text.py` | extract paper text from PDFs | stable mentor corpus ingestion |
| `scripts/normalize_transcripts.py` | clean chats/transcripts | makes spoken feedback reusable |
| `scripts/build_draft_map.py` | split the mature draft into section/paragraph maps | supports targeted diagnostics |
| `scripts/run_term_consistency_check.py` | scan terminology consistency | reduces terminology drift |
| `scripts/build_revision_packet.py` | assemble a revision packet for an agent | stabilizes agent input shape |
| `scripts/diff_revision.py` | generate before/after comparison | enables review and iteration |
| `scripts/export_alignment_scorecard.py` | export mentor-alignment scores | supports evaluation |

Rules:
1. use scripts for deterministic operations whenever possible,
2. never let scripts impersonate mentor judgment,
3. every script needs a clean input/output contract.

## 7.4 Hooks: automation and runtime glue

Hooks have exactly one job:
**trigger scripts or persist state at the right moment.**

They do not reason and they do not rewrite papers.

| Hook | Trigger | Purpose |
|---|---|---|
| `session-bootstrap` | before mentor-brain or polish runs | load latest mentor profile, draft path, and prior artifacts |
| `artifact-persist` | after each lane completes | save outputs into `.omx/` or `artifacts/` |
| `revision-verify` | after each polish pass | run terminology scan, diff export, and risk refresh |
| `notify-long-run` | after long jobs finish | notify the user to return and review |
| `state-checkpoint` | during long analysis | write progress into state/notepad for recovery |

Rules:
1. hooks trigger, not think;
2. hooks touch script/state, not the draft content itself;
3. hooks must remain auditable and rerunnable.

Use these six dimensions for the first round:
- supervisor similarity,
- reduced AI flavor,
- terminology accuracy,
- paragraph rhythm,
- claim precision,
- revision explainability.

### Minimum experiment

Inputs:
- one mature draft,
- 5–20 high-value supervisor materials,

- mentor brain profile,
- xray report,
- revision agenda,

## 12. Relationship to the future full architecture

This version does not reject the full architecture.
It simply **reorders it**.

The full system can still grow toward:
- `lab-dive`
- `story-shape`
- `paper-draft`
- `reviewer-shield`
- `venue-fit`

But the right order is:

> first make mentor-brain + diagnosis + polish good enough to rescue mature drafts, then expand into a full writing operating system.

---

## 13. Final architectural statement

For this project, the correct first product definition is **not**:

> “Let AI automatically write my paper.”

It is:

> “Distill the supervisor’s judgment into a system that guides Codex to inspect, diagnose, and rewrite mature papers.”

If that works, expanding from “rescue a mature draft” to “write a whole paper” becomes natural.
If that does not work, early full-stack automation will only create a larger and less useful system.

So the v0.0.2 architectural conclusion is explicit:

1. `mentor-brain` is the first priority;
2. the second priority is the revision engine composed of `paper-xray + mentor-polish + truth-check-lite`;
3. skill / agent / script / hook must remain strictly layered;
4. every critical stage must emit artifacts;
5. the first real experiment should run on the user’s current mature paper before expanding scope.
