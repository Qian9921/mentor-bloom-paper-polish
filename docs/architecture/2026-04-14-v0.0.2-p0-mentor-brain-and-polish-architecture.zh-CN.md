# IPNL Paper Pilot v0.0.2 P0 架构设计：导师脑 + 修改润色优化检查

状态：P0 重构版 / 建议作为下一轮实现基线
读者：项目负责人、Codex 操作员、后续实现者
范围：只聚焦第一阶段最该落地的两大模块——`mentor-brain` 与 `polish/check engine`
立场：本版明确覆盖你当前最痛的真实场景——“研究已经做完，但成熟论文长期写不到导师要求的层级”

---

## 0. 这版文档为什么要重写

v0.0.1 的大方向没有错，但它还有三个明显问题：

1. **首版范围还是太大。**
   lane 设计是完整的，但对第一阶段来说，真正能最快产生价值的不是全链路写论文，而是先把“成熟稿救出来”。

2. **对 Codex 自身能力的映射还不够细。**
   原文清楚区分了 skill / agent / script，但对 **hook、运行时状态、可验证 artifact、OpenAI Codex 原生能力** 的利用还不够显式。

3. **P0 路径还不够从真实痛点倒推。**
   你的真实痛点不是“完全不会生成论文”，而是：
   - 你有 idea；
   - 你会做实验；
   - 研究很快做完；
   - 但论文改到导师满意极其困难；
   - 导师会指出 AI 味、机器味、术语不准、节奏不对、表述不狠、不像他。

因此，**v0.0.2 直接把第一阶段产品定义为：一个面向成熟论文改稿场景的“导师脑驱动修改系统”**。

一句话说清：

> 先不要贪全自动写整篇论文；先把“导师脑 + 检查/诊断/润色/修订”做成能在现有成熟稿上稳定出结果的系统。

## 1. 真实用户场景与第一成功定义

### 1.1 当前真实场景

用户画像不是泛化“任何研究生”，而是更具体的你当前场景：

- 来自 GNSS / Navigation / Deep Learning / Sensor Fusion 方向实验室；
- 能做研究，也能跑实验；
- 手里已经有一篇相对成熟的稿子；
- 主要瓶颈不是“没内容”，而是“表达上始终达不到导师要求”；
- 导师反馈往往集中在：
  - AI 味重；
  - 术语不准；
  - 句子像在翻译；
  - 段落节奏差；
  - transition 太硬；
  - claim 不够狠或不够稳；
  - 重要信息排序不对；
  - 结论、贡献、novelty 的强调方式不像导师。

### 1.2 第一阶段必须解决什么

第一阶段不是要解决“从零写一篇完整 paper 的所有问题”，而是要解决：

> 给系统一篇成熟论文 + 导师材料 + 你们的聊天/批注/历史修订痕迹，系统能否像一个导师指导者一样，稳定地引导 Codex 找问题、定优先级、改表达、降 AI 味、拉高导师相似度？

### 1.3 第一阶段成功标准

若下面四条成立，就说明 P0 值得继续扩：

1. 系统能从导师 PDF / TXT / 聊天记录 / transcript 中蒸馏出稳定可复用的 **导师写作协议**；
2. 系统能对一篇成熟稿输出有用的 **问题诊断报告**，而不只是泛泛地说“多润色一下”；
3. 系统能按优先级给出 **导师式修改方案**，并生成一版更接近导师要求的文本；
4. 用户主观感受明显改善：从“改三个月很痛苦”变成“能快速定位问题、集中火力修改”。

## 2. P0 范围冻结：只做双核，不做全栈

### 2.1 P0 只做两个核心模块

P0 版本只把下面两个模块做深：

1. **`mentor-brain`**
   - 负责学习导师怎么判断、怎么写、怎么改；
   - 负责沉淀导师术语、导师偏好、导师厌恶表达、导师式 paragraph movement；
   - 它不写当前论文，它只提供“导师智慧”。

2. **`polish/check engine`**
   - 负责吃进一篇成熟稿；
   - 检查 AI 味、用词、节奏、结构、transition、术语、claim/evidence 表达风险；
   - 生成修改 agenda；
   - 驱动 Codex 逐段逐句改到更像导师。

### 2.2 P0 只保留一个轻量支撑脑

P0 不做完整 `lab-dive`，但必须有一个 **`project-context-lite`**：

- 从成熟稿、摘要、图表、方法、结论、已有 notes 中抽出最基本的项目事实；
- 防止 `mentor-polish` 只改语气不顾科学内容；
- 它是导师润色的安全护栏，而不是完整研究系统。

### 2.3 明确不做的事情

P0 明确不做：

- 从零自动生成整篇论文；
- 完整 reviewer simulation；
- 多导师混合蒸馏；
- 持续后台学习；
- 全领域通用学术写作；
- 没有 artifact 的黑盒“一键神改”。

## 3. OpenAI Codex / OMX 在这个项目里到底能做什么

这一节非常关键，因为产品设计必须建立在 **Codex 真正擅长什么** 上，而不是空想。

### 3.1 能力总表

| 能力层 | 当前可用能力 | 在本项目里的正确用法 | P0 是否关键 |
|---|---|---|---|
| 长上下文分析与重写 | 读取长文档、比较版本、综合多来源信息 | 蒸馏导师脑、对照成熟稿、生成修订 agenda | 是 |
| 本地文件与仓库检查 | 读目录、读文件、执行只读 shell、结构化诊断 | 收集导师材料、读取稿件、组装 artifact | 是 |
| 可验证执行 | 运行脚本、生成报告、导出中间文件 | 做 manifest、diff、规则检查、序列化输出 | 是 |
| 多步工作流编排 | 用 skill / plan / state 维持流程一致性 | 让改稿流程可重复、可追踪、可调优 | 是 |
| 多 agent 分工 | planner / architect / executor / critic / verifier 等角色 | 在高复杂度时分拆：诊断、改写、复核 | 次关键 |
| 官方文档与外部检索 | Context7 / web / 官方文档检索 | 查 Codex/SDK/格式规范；查必要写作原则 | 次关键 |
| 持久状态与记忆 | `.omx/state` / notepad / project-memory | 保存当前 paper 实验结论、规则、失败案例 | 是 |
| Hook / 通知 / HUD | omx hooks / tmux-hook / notify / hud | 自动保存 artifact、长任务提醒、状态可见化 | 是 |
| 代码/文本智能检查 | code-intel、shell、regex、diff | 术语一致性、模式扫描、前后版本差异比较 | 是 |

### 3.2 Codex 在本项目最值钱的能力

P0 阶段最值钱的不是“自动写”，而是四件事：

1. **读很多不同来源的导师材料，然后压成一个可执行的判断协议；**
2. **对成熟稿做细粒度诊断，而不是只给空泛建议；**
3. **针对具体句子和段落做重写，并保持全文风格一致；**
4. **把每次修改过程沉淀成 artifact，方便继续迭代。**

### 3.3 Codex 不该被误用的地方

Codex 不该被这样用：

- 不带证据地“猜导师会怎么写”；
- 只做表面润色，不看 claim 是否被改歪；
- 不留中间产物，改完也不知道为什么；
- 把脚本硬堆成“假智能”。

### 3.4 对 P0 最重要的运行时原则

1. **Skill 管流程，不代替判断；**
2. **Agent 做判断、诊断、改写；**
3. **Script 做确定性整理、扫描、导出；**
4. **Hook 只做自动触发，不做推理。**

这四层一旦混掉，整个系统就会很快退化成不可维护的大杂烩。

## 4. P0 的核心系统图：双核 + 两个护栏

### 4.1 系统总览

P0 建议只保留四个显式层：

1. **Mentor Brain Core**
2. **Draft Xray & Revision Engine**
3. **Project Context Lite**
4. **Verification & Artifact Guardrails**

其中：

- `Mentor Brain Core` 负责“导师到底怎么想”；
- `Draft Xray & Revision Engine` 负责“这篇成熟稿现在哪里不行，以及怎么改”；
- `Project Context Lite` 负责“别把科学内容改坏”；
- `Verification & Artifact Guardrails` 负责“全过程可检查、可复盘、可复用”。

### 4.2 Mentor Brain Core

#### 使命
从可信导师材料中提炼出一个 **可被 Codex 执行的导师写作判断系统**。

#### 输入
- 导师已发表论文 PDF / TXT
- 导师修改过的学生论文版本（若有）
- 导师语音转录 / 会议记录 / 聊天记录
- 导师对你论文的批注、批评、常见口头反馈
- 少量同研究线高质量参考论文（只作为补充，不盖过导师）

#### 输出
- `mentor_material_manifest`
- `mentor_corpus_index`
- `mentor_writing_protocol`
- `mentor_rewrite_patterns`
- `mentor_rejection_lexicon`
- `mentor_terminology_preferences`
- `compiled_mentor_profile`

#### 它必须回答的问题
- 导师如何组织一段话？
- 导师最讨厌哪些学生/AI 常见表达？
- 导师如何把弱句改成强句？
- 导师如何收束贡献、abstract、conclusion？
- 导师偏好的术语、语气、力度、节奏是什么？

#### 它绝对不该做的事
- 不研究当前项目真假；
- 不直接改当前稿子；
- 不拿当前草稿反向训练导师脑。

### 4.3 Draft Xray & Revision Engine

#### 使命
对一篇成熟稿做“像导师一样的体检 + 像导师一样的改稿”。

#### 内部子模块
1. `draft-intake`
2. `paper-xray`
3. `issue-ranking`
4. `truth-check-lite`
5. `mentor-polish`
6. `revision-verify`

#### 它的核心问题
- 这篇稿子最像 AI 的地方在哪里？
- 哪些地方其实不是“语言不好”，而是“贡献讲法不对”？
- 哪些句子弱、虚、模糊、没击中重点？
- 哪些段落像在堆信息而不是讲故事？
- 哪些术语、转折、对比、强调方式不符合导师习惯？
- 改完以后，哪些地方真的变好了，哪些只是换了说法？

### 4.4 Project Context Lite

#### 使命
作为“轻量项目脑”，在不重做完整 `lab-dive` 的前提下保护 scientific correctness。

#### 最小输入
- mature draft
- abstract / intro / method / experiment / conclusion
- 图表标题与 caption
- 关键公式 / 算法描述
- 用户提供的补充 notes

#### 最小输出
- `project_fact_sheet`
- `claim_evidence_minimap`
- `terminology_ground_truth`
- `do_not_break_constraints`

### 4.5 Verification & Artifact Guardrails

#### 使命
保证每轮修改都不是黑盒。

#### 需要强制产出的 artifact
- 输入材料 manifest
- 诊断报告
- 修改 agenda
- 修改前后 diff
- 未解决风险清单
- 下一轮建议

## 5. 双脑边界：什么能学，什么不能学

### 5.1 导师脑允许学习的材料

允许进入 `mentor-brain` 的材料：

- 导师本人写的论文；
- 导师本人说过的话（聊天、语音、批注、回复）；
- 导师明确改过的文字；
- 导师认可的表达范式；
- 与导师研究线一致的少量强参考文献。

### 5.2 导师脑禁止学习的材料

禁止进入 `mentor-brain` 的材料：

- 当前正在修改的目标论文正文；
- 学生自己写得很烂的初稿；
- 来路不明的 AI 文本；
- 与导师风格明显不一致但仅仅“看起来很高级”的模板话术。

### 5.3 项目脑轻量上下文允许包含什么

`project-context-lite` 允许吸收：

- 稿件中的核心事实；
- 图表、实验设置、数据来源；
- 用户补充的真实研究解释；
- 关键 claim 与 evidence 的绑定关系。

### 5.4 为什么边界必须死守

因为你现在最需要的不是“再来一个会写套话的 AI”，而是：

> 一个能够稳定区分“导师风格”和“当前项目事实”的系统。

这两个东西一旦混掉，系统就会同时变得：

- 不像导师；
- 也不忠于项目；
- 改得越多越脏。

## 6. 按你当前痛点倒推的 P0 工作流

### 6.1 Stage A：导师材料摄入与清洗

目标：先把导师材料整理干净。

输入类型按优先级排序：

1. 导师亲自写的论文；
2. 导师对你或他人论文的修改痕迹；
3. 导师聊天记录、语音转录、评语；
4. 高质量参考论文。

输出：

- 文档清单；
- 来源等级；
- 文本抽取结果；
- 可供蒸馏的标准化材料包。

### 6.2 Stage B：导师脑蒸馏

目标：从材料里提炼“导师写作协议”。

核心产物至少要有：

- 常用术语表；
- 常见否定表达 / 讨厌表达；
- introduction / method / experiment / conclusion 的偏好模式；
- 句级改写模式；
- 段级组织模式；
- claim 强化与收束模式。

### 6.3 Stage C：成熟稿 intake

目标：不要一上来就乱改，先把稿件拆干净。

至少抽出：

- 章节结构；
- 标题和摘要；
- 关键贡献句；
- 关键图表；
- 方法核心段；
- 实验结论段；
- conclusion；
- 当前最像 AI 的热点区域。

### 6.4 Stage D：Paper Xray 诊断

`paper-xray` 不是“泛泛提意见”，而是要按维度出报告。

建议第一版固定八个维度：

1. `mentor-drift`：不像导师；
2. `ai-smell`：机械、均质、模板感；
3. `terminology-risk`：术语不稳、不准、不像本领域；
4. `claim-precision`：弱 claim、虚 claim、范围不稳；
5. `transition-rhythm`：段间、句间衔接不自然；
6. `paragraph-architecture`：段内信息排序差；
7. `evidence-language-gap`：结果明明不错，但语言没把价值说出来；
8. `truth-risk-lite`：润色可能会把科学含义改歪的地方。

输出：

- `xray_report`
- `issue_board`
- `severity_ranking`
- `rewrite_candidates`

### 6.5 Stage E：Revision Agenda 生成

不是发现 100 个问题就 100 个都一起改，而是要排序。

建议优先级规则：

1. 先改 **会伤 scientific meaning** 的问题；
2. 再改 **导师一眼会皱眉** 的问题；
3. 再改 **最像 AI** 的问题；
4. 最后改一般润色问题。

输出：

- `revision_agenda.md`
- `section_priority_map.json`
- `do_first_do_not_touch.md`

### 6.6 Stage F：Mentor Polish 执行

润色必须分层做，不能一锅端。

建议固定四层：

1. **Macro pass**：标题、摘要、贡献句、结论收束；
2. **Paragraph pass**：段落功能、信息排序、主题句、结尾句；
3. **Sentence pass**：措辞、力度、压缩、精确性；
4. **Terminology pass**：术语、缩写、表达统一。

### 6.7 Stage G：修改后验证

每次改完必须回答：

- 哪些问题被解决了？
- 哪些问题只换了表面说法？
- 哪些修改有 scientific meaning 漂移风险？
- 哪些地方需要你本人或导师最终拍板？

输出：

- `revision_delta_report`
- `remaining_risks`
- `mentor_alignment_scorecard`

## 7. Skill / Agent / Script / Hook 的详细分工

这是本版最重要的结构化重写部分。

## 7.1 Skill：流程与治理层

Skill 负责定义：

- 什么时候进入这个 lane；
- 入口材料要求；
- 输出 artifact 合同；
- 失败重试逻辑；
- handoff 给谁；
- done criteria。

### P0 推荐 Skills

| Skill 名 | 角色 | 输入 | 输出 | 是否 P0 必做 |
|---|---|---|---|---|
| `mentor-brain` | 导师材料学习与蒸馏总入口 | PDF/TXT/chat/transcript | mentor profile / protocol / lexicon | 是 |
| `paper-xray` | 成熟稿诊断入口 | mature draft + mentor profile | xray report / issue board | 是 |
| `mentor-polish` | 导师式修订入口 | mature draft + revision agenda + mentor profile | revised draft / revision notes | 是 |
| `truth-check-lite` | 科学含义保护入口 | revised draft + fact sheet | do-not-break warnings / lite verification | 是 |
| `reference-mirror` | 强参考镜像比对 | weak paragraph + strong refs | contrast packet | 建议做 |
| `draft-compare` | 前后版本差异解释 | before / after drafts | delta report | 建议做 |

### Skill 设计原则

1. Skill 只写 **流程规则**，不把重写判断写死；
2. Skill 必须要求输出 artifact；
3. Skill 必须显式说明何时不能继续自动改；
4. Skill 必须优先服务成熟稿改稿场景，而不是泛化大而全流程。

## 7.2 Agent：真正做思考、诊断、改写的人

P0 阶段不需要造很多“新神秘 agent 名字”，而是应该尽量复用当前 Codex 原生分工。

### 当前环境里最适合本项目的 agent 类型

| Agent 类型 | 在本项目中的职责 | P0 用法 |
|---|---|---|
| `planner` | 规划一次改稿任务、排序问题、安排 pass 顺序 | 用来做 revision agenda |
| `architect` | 维护边界，避免导师脑/项目脑/润色层混乱 | 用来审结构与 guardrails |
| `executor` | 实际改句子、改段落、改结构 | P0 主力 |
| `critic` | 从“导师会不满意哪里”角度挑刺 | 用来挑战 AI 味和空泛表达 |
| `verifier` | 检查是否真的修好、有没有改歪 | 每轮 polish 后复核 |
| `researcher` | 查必要外部写作原则或术语背景 | 低频辅助 |
| `writer` | 把修改说明、artifact 摘要写清楚 | 生成面向用户的解释 |

### P0 推荐 Agent 运行模式

默认单 leader 即可；复杂时再分三步：

1. `critic` 先出问题；
2. `executor` 根据 mentor brain 改；
3. `verifier` 检查有没有改坏。

### Agent 不该被滥用的地方

- 不要让 agent 自己偷偷定义 artifact 结构；
- 不要让 agent 在没有 mentor profile 的情况下装作“像导师”；
- 不要让多个 agent 同时改同一段而没有 merge 规则。

## 7.3 Script：确定性工位层

Script 不做思考，只做稳定工位。

### P0 推荐脚本能力

| Script / 目录建议 | 作用 | 为什么需要 |
|---|---|---|
| `scripts/build_mentor_manifest.py` | 为导师材料生成清单与来源等级 | 防止材料混乱 |
| `scripts/extract_pdf_text.py` | 提取 PDF 文本 | 让后续蒸馏稳定 |
| `scripts/normalize_transcripts.py` | 清理聊天/转录文本 | 让导师语料可用 |
| `scripts/build_draft_map.py` | 把 mature draft 拆成 section / paragraph map | 支持按段诊断 |
| `scripts/run_term_consistency_check.py` | 做术语一致性扫描 | 降低术语飘移 |
| `scripts/build_revision_packet.py` | 组装一次改稿需要的 packet | 让 agent 输入稳定 |
| `scripts/diff_revision.py` | 生成 before/after 差异摘要 | 便于复盘 |
| `scripts/export_alignment_scorecard.py` | 导出导师对齐评分表 | 支持评估 |

### Script 设计原则

1. 能 deterministic 就别交给 agent 瞎猜；
2. 但 script 不能篡位去做“导师式判断”；
3. 每个 script 都要有清晰输入输出文件。

## 7.4 Hook：自动触发与运行时粘合层

Hook 的职责只有一个：**在合适时机自动触发脚本或保存状态**。

它不做思考，不做改写，不做风格判断。

### P0 推荐 Hooks

| Hook 名 | 触发时机 | 作用 |
|---|---|---|
| `session-bootstrap` | 开始一次 mentor-brain / polish 任务前 | 自动加载最新 mentor profile 路径、草稿路径、上轮 artifact |
| `artifact-persist` | 每次 lane 完成后 | 自动把输出写入 `.omx/` 或 `artifacts/` |
| `revision-verify` | 每轮 polish 结束后 | 触发术语扫描、diff 导出、风险清单刷新 |
| `notify-long-run` | 长时间任务完成时 | 通知用户该回来 review |
| `state-checkpoint` | 大文档分析中间阶段 | 把当前进度写到 notepad/state，防止上下文损失 |

### Hook 设计原则

1. Hook 只触发，不推理；
2. Hook 只调 script / state，不直接改论文；
3. Hook 必须可关闭、可重跑、可审计。

## 10. P0 的验收标准：不是“改了”，而是“改得像导师且更稳”

### 10.1 核心验收维度

第一版建议固定用下面六个维度验收：

| 维度 | 要看什么 |
|---|---|
| 导师相似度 | 是否更像导师会写/会改的样子 |
| AI 味下降 | 是否减少模板腔、均质腔、翻译腔 |
| 术语准确性 | 是否更贴近 GNSS / navigation / DL 实验室语境 |
| 段落节奏 | 段落是否更有推进感、信息排序更合理 |
| claim 精确性 | 结论、贡献、比较是否更稳更准 |
| 可解释性 | 系统是否能解释“为什么这样改” |

### 10.2 最小实验方案

第一轮实验建议就用你现在这篇成熟稿：

输入：
- 1 篇成熟 draft；
- 5–20 份高价值导师材料；
- 你和导师的若干聊天/批注/转录文本；
- 一小批本方向高质量参考文献。

输出：
- 一份导师脑画像；
- 一份 xray 诊断报告；
- 一份 revision agenda；
- 一版改后稿件；
- 一份前后差异和风险报告。

如果这套实验能明显帮你缩短改稿痛苦，P0 就算成功。

### Phase 1：文档与 artifact 合同先定死

## 12A. 未来形态建议：三脑并行，而不是一脑包打天下

你现在提出的方向，我认为是对的，而且从长期看比“双脑”更稳：

未来更合理的终局形态不是一个混合大脑，而是：

1. **导师脑（Mentor Brain）**
2. **项目脑（Project Brain）**
3. **专家脑（Expert Brain）**

但我还想补一个更关键的建议：

> 最好不是“只有三脑”，而是 **三脑 + 一层编排/裁判层**。

也就是说，真正稳定的长期结构更像：

- `mentor-brain`
- `project-brain`
- `expert-brain`
- `orchestrator / judge layer`

其中三脑负责提供不同类型的智慧，
而编排层负责判断：

- 当前问题该问哪一个脑；
- 哪两个脑应该一起参与；
- 哪个脑有权最终约束改稿方向。

### 为什么我支持三脑并行

因为这三个脑本质上在回答完全不同的问题：

#### 1. 导师脑

它回答的是：

- 导师怎么写？
- 导师怎么改？
- 导师怎么组织 paragraph / results / discussion？
- 导师最反感哪些表达？
- 导师会如何 tighten、收束、命名、转折？

它负责的是：

> **写作智慧、改稿智慧、表达判断、叙述技法**

#### 2. 项目脑

它回答的是：

- 你的 idea 是什么？
- 你的代码到底做了什么？
- 你的实验设置、消融、结果、失败点是什么？
- 哪些 claim 真的被你的实验支撑？
- 哪些图、表、指标、比较是你这个项目独有的真实事实？

它负责的是：

> **当前研究项目的事实、证据、边界、实现细节**

#### 3. 专家脑

它回答的是：

- 这个方向的算法、数学、理论、创新点应该怎样理解？
- 这个方法和已有工作在技术上到底差在哪？
- 这个证明、这个建模、这个 estimator、这个 loss 应该怎么讲才专业？
- 哪些论文是真正高质量、值得借鉴的专家级样例？

它负责的是：

> **外部专家知识、文献知识、算法/数学/创新/技术判断**

### 我额外补的建议：一定要有编排/裁判层

因为很多问题都不是单脑能答的。

例如：

- “这段 results 怎么改得像导师？”  
  主要靠 **导师脑 + 项目脑**

- “这段方法论是不是技术上讲歪了？”  
  主要靠 **项目脑 + 专家脑**

- “这个 contribution sentence 该怎么写，既像导师又不 overclaim？”  
  需要 **导师脑 + 项目脑 + 专家脑** 一起约束

所以长期来看，我更推荐把未来架构定义成：

> **三脑并行 + 一个负责任务路由、冲突消解、最终裁决的编排层**

---

## 12B. 三脑之间必须死守边界

如果你真的走三脑并行，这些边界必须写死。

### 导师脑的职责边界

导师脑 **只负责**：

- 写作智慧
- 改稿智慧
- 段落推进
- figure/table narration
- 术语使用偏好
- 风格判断
- revision operator

导师脑 **绝不负责**：

- 算法是否先进
- 数学是否正确
- method 是否成立
- 实验是否充分
- technical novelty 是否真的存在

也就是说：

> 导师脑是 **写作/改稿的专家**，不是算法/数学的专家。

### 项目脑的职责边界

项目脑 **只负责**：

- 当前项目的真实事实
- idea、代码、实验、结果、图表、失败点
- claim / evidence 对齐
- 当前稿件中的真实研究细节

项目脑 **绝不负责**：

- 生成通用写作偏好
- 影响导师脑的风格学习
- 充当领域理论标准答案

### 专家脑的职责边界

专家脑 **只负责**：

- 外部论文知识
- 算法/数学/理论/创新判断
- 领域惯例
- 专家级技术表达

专家脑 **绝不负责**：

- 模仿导师风格
- 决定导师会怎么改句子
- 污染当前项目事实

## 12C. 我对你这个想法的进一步补充

如果只给建议，我会在你的“三脑并行”之上再补三点：

### 1. 三脑不要同时直接改稿

更好的模式是：

- 三脑各自产生 judgment
- 编排层汇总
- 最后由具体的执行 agent 改稿

否则很容易出现：

- 专家脑把段落改得很专业但不像导师；
- 导师脑把段落改得很像导师但 technical nuance 不够；
- 项目脑又坚持本项目事实，三者互相拉扯。

### 2. 三脑都应该输出 artifact，而不是隐性记忆

未来建议每个脑都有自己的可审计 artifact：

- `mentor_profile`
- `project_dossier`
- `expert_reference_synthesis`

这样 downstream agent 才知道自己是在消费谁。

### 3. 未来可以有一个 “cross-brain synthesis” artifact

某些最难的问题，不应该直接改稿，而是先生成：

- `cross_brain_decision_note`

里面明确写：

- 导师脑怎么想
- 项目脑怎么想
- 专家脑怎么想
- 编排层最后为什么这么定

这会极大提升系统透明度和可解释性。

## 12D. 但当前阶段必须保持冻结：先把导师脑做到最好

虽然我支持未来三脑并行，
但我同样强烈支持你现在的判断：

> **当前不要同时展开三脑实现，先把导师脑做到最好最稳最可用。**

原因很简单：

1. 你的当前最大痛点仍然是“稿子不像导师”；
2. 导师脑是当前最短价值路径；
3. 如果导师脑还没成型，就同时做项目脑/专家脑，系统会快速扩散；
4. 三脑并行的前提，是至少先有一个脑做得足够强，能成为标准样板。

所以我建议文档上明确写：

### 当前冻结策略

- **P0 / P0+：只重点做导师脑**
- 项目脑和专家脑只保留未来接口与边界设计
- 暂不实现完整学习/消费/编排执行层

也就是说：

> 三脑是未来总架构；  
> 导师脑是当前唯一主战场。

并且，就当前特定阶段，还可以再进一步收窄成：

> **导师脑 + mentor-only polish mode**

也就是说，当前先默认：

- 论文主要 scientific conclusion 已正确；
- 当前不引入项目脑的深度裁决；
- 当前只从导师写作技法与改稿技法优化稿件。

这会让当前阶段的架构更聚焦，也更符合真实收益路径。

## 13. 最终架构宣言

对于这个项目，第一阶段最正确的产品定义不是：

> “让 AI 自动帮我写论文。”

而是：

> “把导师的判断方式蒸馏出来，变成一个能指导 Codex 检查、诊断、重写成熟论文的系统。”

如果这件事做成了，后面从“救成熟稿”扩到“写整篇论文”，才是顺水推舟。

如果这件事没做成，过早追求全自动写稿，只会把系统做得更大、更虚、更难用。

所以，**v0.0.2 的架构结论非常明确：**

1. 第一优先级是 `mentor-brain`；
2. 第二优先级是 `paper-xray + mentor-polish + truth-check-lite` 组成的修改引擎；
3. Skill / Agent / Script / Hook 必须严格分层；
4. 所有关键阶段都必须产出 artifact；
5. 先在你的成熟论文上做实验，跑通闭环，再扩完整系统。
